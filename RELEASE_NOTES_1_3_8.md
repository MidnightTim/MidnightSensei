# Midnight Sensei v1.3.8 — Shaman Pass, Spell/Talent Snapshots, Debug Overhaul & Level Gate

## Summary

1.3.8 delivers a full Midnight 12.0 spec pass for Elemental and Restoration Shaman, correcting multiple spell IDs and removing abilities pruned from the game. A persistent spell and talent snapshot system is introduced for contributor tooling, replacing live API enumeration with a reliable per-login store. The debug window is reorganised with distinct sections, the X button rendering bug is fixed, and a level 80 gate prevents low-level content from polluting grade history.

---

## Level Gate

Fights recorded below player level 80 are now silently discarded — no history entry, no leaderboard broadcast. This prevents leveling dungeon kills, old-world farming sessions, and timewalking alts from appearing in grade history or weekly averages.

The gate sits immediately after the minimum fight duration check in `OnCombatEnd`, using `UnitLevel("player")`. The HUD still shows the fight result locally for the session but nothing is persisted.

## Spell & Talent Snapshot System

A persistent snapshot system captures the player's full spellbook and active talent tree automatically, storing the results in `CharDB` so export tools read from reliable persisted data rather than making live API calls at the moment the button is pressed.

### When snapshots are built
- **Login** — 2 second delay after `SESSION_READY` to allow spellbook to fully load
- **Spec change** — `PLAYER_SPECIALIZATION_CHANGED` or `PLAYER_TALENT_UPDATE`, debounced 1 second
- **Spell learn** — `SPELLS_CHANGED`, debounced 1 second (fires on login, talent changes, learning new spells)

### What is stored
**`CharDB.spellSnapshot`**
```
{ timestamp, specName, className, spells[] }
spells[i] = { spellID, name, subName }
```
Enumerated via `C_SpellBook.GetSpellBookItemInfo` / `C_SpellBook.GetSpellBookSkillLineInfo`. Sorted by spellID, matching the format of the uploaded spec spell list files for direct diff comparison.

**`CharDB.talentSnapshot`**
```
{ timestamp, specName, className, talents[] }
talents[i] = { spellID, nodeID, name, rank }
```
Enumerated via `C_Traits` tree walk. Sorted by nodeID, matching the format of the uploaded talent files.

Both snapshots include a timestamp and the spec they were captured under so you know exactly which build and when.

### First session
If no snapshot exists yet (brand new install or character), the export commands print a clear message directing to `/reload`. After reload, snapshots are built automatically within 2 seconds of login.

## Debug Tools Window

### X Button Fix
The close button was rendering as a box (□) due to the `✕` Unicode character being outside FRIZQT__.TTF's basic Latin coverage — a known constraint documented in the project handoff. Replaced with plain `"X"` which renders correctly. Title text anchor also corrected to leave room for the button.

### Section Reorganisation

**Before:** one "Recovery Tools" section containing Talent Export, Spells Export, Backfill M+ Keys, and Clean Payload.

**After:** two distinct sections:

**Class Debugging** (accent/cyan separator) — contributor tooling:
- Talent Export
- Spells Export

**Recovery Tools** (orange separator) — operational recovery:
- Backfill M+ Keys
- Clean Payload

A reusable `AddSectionLabel(text, color)` helper was added so future sections can be added with one line.

## Grade History

The **All Characters** filter tab is removed. Since the schema v3 migration in 1.3.2, encounters are stored in `SavedVariablesPerCharacter` — a separate DB table per character that is only loaded for the current character. The "All Characters" filter was silently showing only the current character's data, identical to "This Character". The tab has been removed rather than repaired since reverting to account-wide encounter storage would re-introduce the cross-character data bleed that motivated the schema v3 split.

## Technical Notes

- `SPELLS_CHANGED` registered as a new event in `Core.lua`
- `Core.ScheduleSnapshots()` exposed on the Core table for event dispatch
- Snapshot builds are fully wrapped in `pcall` — API failures on any node are skipped silently
- `CharDB.spellSnapshot` and `CharDB.talentSnapshot` are `nil` until first built — no SavedVariables bloat on fresh installs
- `Core.VERSION` fallback updated to `"1.3.8"`
